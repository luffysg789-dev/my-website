# P-Mining 原型设计稿

日期：2026-03-28

## 目标

在现有项目中新增一个移动端优先的网页挖矿原型 `P-Mining`，页面风格严格参考用户提供的 UI 稿，并保留少量适配移动网页的微调空间。

本次目标不是接入真实 Firebase，而是先做一个高保真前端原型，重点验证：

- 深色玻璃拟态视觉是否成立
- 挖矿主流程是否顺滑
- 邀请、记录、个人页的信息结构是否清晰
- Nexa Pay 宿主内授权登录的产品形态是否成立

## 成功标准

- 游戏大全新增 `P-Mining` 入口卡
- 用户可从独立路由进入 `P-Mining`
- 首页可展示余额、算力、预计收益、圆环倒计时领取区
- 底部导航可切换 `挖矿 / 邀请 / 记录 / 我的`
- 原型默认运行在 `Nexa Pay` 宿主授权登录场景下
- 个人数据刷新后保留
- 全网统计有实时变化的模拟感
- 挖矿领取遵循 60 秒冷却和正式奖励公式
- 页面整体视觉贴近用户提供的参考 UI

## 范围边界

### 本次包含

- `P-Mining` 独立页面与游戏入口接入
- 参考图风格的高保真移动端界面
- 四个主页面的完整浏览与交互
- 本地持久化的个人数据
- 模拟实时流的全网数据
- 60 秒领取倒计时与奖励结算
- 邀请码展示、复制、绑定与防自绑校验
- 记录列表与分类切换
- “我的”页面中的用户资料与规则说明
- 宿主授权态的前端模拟抽象

### 本次不包含

- 真实 Firebase Auth
- 真实 Firestore / Transaction / onSnapshot
- 多端实时同步
- 真正跨账号的邀请码关系网络
- 服务端反作弊
- 提现、充值、链上资产流转

## 产品前提

`P-Mining` 只存在于 `Nexa Pay` 内，不做独立站登录页。用户身份由 `Nexa Pay` 宿主完成授权，游戏页面只消费宿主提供的登录结果。

这意味着原型页面默认站在“已登录”之后：

- 能拿到用户 UID
- 能拿到邮箱或昵称
- 能拿到宿主连接状态
- 不额外提供站内注册登录页

## 入口与路由

### 游戏大全入口

在 `public/games-config.js` 中新增一个默认游戏：

- `slug`: `p-mining`
- `name`: `P-Mining`
- `description`: 强调云挖矿、透明分配、邀请加成与移动端体验
- `route`: `/p-mining/`
- `icon`: 使用镐头意象或与用户图稿一致的品牌图形
- `actionText`: `开始挖矿`

### 页面路由

推荐保持和现有独立游戏一致的静态目录结构：

- `public/p-mining/index.html`
- `public/p-mining/style.css`
- `public/p-mining/script.js`

## 视觉方向

### 参考原则

视觉直接以用户提供的 UI 作为基准，不再重新探索风格方向。

实现时允许做少量网页化微调，但必须满足：

- 第一眼仍然是同一套产品
- 卡片层级、主色、结构顺序不跑偏
- 用户一看就能认出与参考图一致

## 风格关键词

- 深色模式
- 玻璃拟态
- 稳定
- 精确
- 金融感
- 挖矿仪式感

## 配色

- 页面主背景：纯黑到深灰过渡
- 卡片背景：深灰半透明
- 卡片边框：低对比半透明白
- 主强调色：橙色 `#F27D26`
- 算力强调色：亮黄
- 主文字：纯白
- 次级文字：半透明白或中灰
- 连接状态：绿色状态点

## 排版

- 标题与大数字：粗体、紧凑、无衬线
- 数据与倒计时：等宽字体
- 标签说明：浅灰、低权重
- 所有积分和算力数字前端统一保留 1 位小数

## 页面结构

## 1. 挖矿页

挖矿页是产品首页，也是主视觉中心。

### 顶部状态区

- 左侧：品牌图标 + `P-Mining`
- 右侧：网络连接状态胶囊
- 状态文案默认显示为“网络已连接”

### 资产主卡

主卡复刻用户给出的结构：

- 标题：`当前持有 P`
- 大号余额数字
- 卡片背景中可放低对比品牌镐头图形
- 下方并列展示：
  - 当前算力
  - 预计收益 / 分

### 领取区

页面中部是大号圆环按钮：

- 圆环外圈展示 60 秒进度
- 中心展示时钟图标与倒计时
- 可领取时展示 `点击领取`
- 冷却中展示 `00:00:xx`
- 点击后给出轻微缩放与辉光反馈

### 全网统计区

圆环按钮下方是统计卡网格，内容参考用户图稿：

- 全网用户
- 已挖出总量
- 今日全网产出
- 全网今日算力
- 剩余总量
- 当前减半周期
- 预计下次减半
- 总发行量
- 预计挖完时间

### 规则说明卡

统计卡之后展示规则提示卡，解释：

- P 是 Nexa 生态内积分系统
- 当前阶段以积分流通与激励为主
- 用户量达到阈值后再进入升级阶段

这张卡保留图稿里的信息性、低干扰定位。

## 2. 邀请页

邀请页由三个主要区域构成：

### 我的邀请码卡

- 标题：`邀请好友`
- 展示个人邀请码
- 右侧是一键复制按钮
- 下方说明“双向各加 10 算力”

### 绑定邀请码卡

- 输入框填写好友邀请码
- 右侧提交按钮
- 已绑定后切为不可编辑状态

### 邀请统计卡

- 已邀请人数
- 邀请算力加成

## 3. 记录页

记录页保持图稿中的顶部分类切换：

- `领取记录`
- `邀请记录`
- `算力变动`

下方列表项采用统一卡片样式：

- 左侧：事件名称、时间
- 右侧：数值变化
- 领取记录附带当时算力
- 邀请记录附带好友邀请码或匿名标识
- 算力变动附带原因说明

## 4. 我的页

我的页展示宿主授权登录下的用户资料：

- 顶部大卡显示头像占位、邮箱或昵称、UID
- 下方信息卡展示：
  - 当前总积分
  - 当前算力
  - 绑定状态
- 底部提供退出登录按钮

原型里“退出登录”表现为回到一个受限占位态或清除本地个人缓存，不模拟完整 Nexa 宿主跳转。

## 底部导航

底部导航固定悬浮在页面底部，保持用户图稿中的四个项目：

- 挖矿
- 邀请
- 记录
- 我的

要求：

- 当前项使用橙色高亮
- 未选中项使用灰色
- 保留玻璃拟态模糊背景
- 处理手机安全区 inset

## 原型数据模型

原型数据分为“个人持久化数据”和“全网模拟实时数据”两层。

### 个人持久化数据

使用 `localStorage` 保存，刷新后不丢失：

- `balance`
- `power`
- `inviteCode`
- `boundInviteCode`
- `inviteCount`
- `invitePowerBonus`
- `claimRecords`
- `inviteRecords`
- `powerChanges`
- `lastClaimAt`
- `createdAt`

默认初始化：

- 初始算力：`10.0`
- 初始余额：`0.0`

### 宿主用户数据

通过模拟的 `Nexa Pay` bridge 注入：

- `uid`
- `email` 或 `nickname`
- `networkConnected`

实现上应封装成一个独立读取层，未来可替换为真实宿主对象。

### 全网模拟实时数据

由前端 ticker 定时推进，不写回服务端：

- `totalUsers`
- `totalMined`
- `todayMined`
- `todayPower`
- `remainingSupply`
- `currentHalvingCycle`
- `nextHalvingDate`
- `estimatedFinishYears`

这些数据在页面运行时按设定速率缓慢变化，制造“网络正在同步”的感觉。

## 核心挖矿逻辑

## 供应与减半

原型中保持正式规则文案和计算参数：

- 总供应量：`2100 亿 P`
- 每 4 年减半一次
- 总周期：`100 年`
- 当前阶段每日上限：`71,917,808 P`

### 奖励公式

单次领取奖励使用正式公式：

`(个人算力 / 全网总算力) × (每日上限 / 1440)`

其中：

- 个人算力来自本地持久化用户数据
- 全网总算力来自模拟实时流

### 领取限制

- 每 60 秒最多领取 1 次
- 冷却中按钮不可重复触发
- 前端保留严格的 `isProcessing` 锁

### 稀缺性模型

即使是原型，也保留以下文案与统计逻辑：

- 当日未挖出的产量不消失
- 仍保留在总量池中
- 页面中的剩余总量随累计挖出量递减

## 邀请与算力逻辑

### 绑定规则

- 每个账号只能绑定一次邀请码
- 不能绑定自己的邀请码
- 绑定成功后双方各增加 `10.0` 算力

### 原型实现方式

因为没有真实后端，多账号关系采用“宿主用户 + 本地演示邀请码池”的方式模拟：

- 当前用户始终拥有自己的邀请码
- 绑定时校验不能输入自己的邀请码
- 系统内置少量演示邀请码作为可绑定对象
- 绑定成功后，当前账号得到 `+10.0` 算力，并记录“好友也已增加 10 算力”的提示文案

这可以先把完整交互与防自绑体验跑通，后续再替换成真实 Firestore 事务。

## 状态与交互细节

## 启动流程

页面启动时：

1. 读取 Nexa 宿主模拟用户
2. 读取本地持久化个人数据
3. 启动全网数据模拟 ticker
4. 恢复倒计时状态
5. 渲染默认首页

## 倒计时恢复

倒计时不能因为切 tab 或刷新而重置。

实现方式：

- 记录 `lastClaimAt`
- 每次渲染根据当前时间反推出剩余秒数
- 如果已超过 60 秒，则自动恢复为可领取状态

## 记录写入

### 领取记录

每次成功领取后写入：

- 数量
- 时间
- 当时算力

### 邀请记录

每次邀请码绑定成功后写入：

- 事件时间
- 绑定的邀请码
- 奖励说明

### 算力变动

每次算力变化都单独写入原因：

- 初始注册
- 邀请奖励
- 未来预留活动奖励

## 错误态设计

虽然是原型，仍需提供清晰的错误提示：

- 宿主授权信息缺失
- 网络连接中断
- 倒计时未结束重复点击
- 邀请码为空
- 绑定自己邀请码
- 已经绑定过邀请码
- 邀请码无效

提示形式以轻量 toast 或卡片内错误文案为主，不使用浏览器原生 `alert`。

## 响应式与移动端适配

页面优先面向手机纵向使用。

### 移动端要求

- 主内容宽度贴近图稿
- 底部导航固定且不遮挡内容
- 长数字支持缩放、换行控制或局部裁切
- 卡片圆角、内边距保持一致

### 桌面端要求

桌面端不重做新布局，只做居中容器与更舒展的留白，让页面看起来像一个完整的移动产品预览。

## 技术落地约束

用户期望的正式技术栈是：

- React + Vite
- Tailwind CSS
- Lucide React
- Framer Motion
- Firebase

但当前仓库是已有静态游戏目录结构，并不是 React/Vite 工程。因此本次原型的落地原则是：

- 先遵循当前仓库结构完成原型接入
- 交互和视觉上尽量对齐未来正式版
- 代码按模块化方式组织，给后续 React 化保留迁移空间

正式版里需要具备但本次只做接口预留的能力：

- Firestore Transaction
- onSnapshot
- 宿主真实授权桥接
- Firestore 权限错误处理

## 测试与验证重点

至少覆盖以下验证点：

- 游戏大全正确出现 `P-Mining` 入口
- 路由可访问
- 默认渲染四个 tab
- 倒计时冷却逻辑正确
- 页面刷新后个人余额和记录仍存在
- 邀请码不能绑定自己
- 邀请码只能绑定一次
- 记录页分类切换可用
- 我的页展示宿主用户信息

## 后续正式版迁移建议

当前原型完成后，下一阶段再切入真实架构时，优先替换以下模块：

1. 宿主用户读取层替换为真实 Nexa 授权桥接
2. 本地个人数据层替换为 Firebase 用户数据
3. 全网模拟 ticker 替换为 Firestore 实时快照
4. 邀请绑定与领取结算替换为事务逻辑

这样可以避免推翻 UI 层，先保住页面与交互成果。
